## Stage 1 — Audit: Production SLM Labeling

**Finding:** The repository contains no production SLM inference. Every result labeled "SLM" in
the benchmark artifacts and summary tables is either deterministic Python or a Git LFS pointer
to weights that have never been loaded.

Key facts:
- Four HuggingFace checkpoint `model.safetensors` files in `models/huggingface/` are Git LFS
  pointers (135 bytes each, ~1.4–2.0 GB not downloaded).
- `subroutine_models.py` routes `search_query`, `search_rank`, and `trace_localize` to
  deterministic rule functions regardless of mode; there is no code path that loads these HF
  weights in the current runtime.
- The `slm_only` and `schema_slm_fallback` labels in the stored benchmark JSONL refer to
  `_slm_like_json_repair()` — pure Python `ast.literal_eval` + regex, zero model inference.
- The stored `gpu_memory_mb: 567` in `artifacts/results/raw/subroutine_benchmark.jsonl` is
  from the original Lambda A10 run (where `failure_classifier_v1` and `patch_risk_classifier_v1`
  were loaded via env var); it cannot be reproduced on this machine.

**Commit:** n/a (audit stage, no code changes)

## Stage 2 — failure_classifier_v1: Real Inference on A100

**Finding:** The checkpoint contains real BERT-tiny weights (17 MB safetensors), but produces
near-uniform logits on real-world failure logs. It should not be deployed.

Evidence:
- 10 hand-written failure log snippets (non-synthetic) run on A100.
- Model confidence: 0.128–0.163 across all classes (random baseline for 9 classes = 0.111).
- Model accuracy: 2/10. Deterministic rules accuracy: 8/10 on the same inputs.
- Root cause: training and eval data both generated by `make_synthetic_examples()` in
  `scripts/train_remaining_subroutines.py:364–403`. The factory injects the exact keyword
  strings the regex rules match. 1.0 dev accuracy is in-distribution memorization, not
  generalization.
- Training summary claims `device: cuda, elapsed: 10.4s, accuracy: 1.0` — the accuracy is a
  tautology of the data construction.

**Commit:** `964ed0b` — "finding: failure_classifier_v1 does not generalize beyond synthetic training data"  
**Analysis file:** `results/failure_classifier_v1_analysis.md`

## Stages 3–4 — search_query_gen: Inference Setup Bug Found and Fixed

**Finding:** Stage 3 used the wrong prompt format (chat template); Stage 4 used the correct
completion format. The model is not globally degenerate, but it is still non-functional on
novel queries.

Stage 3 (wrong format — `apply_chat_template` with system/user roles):
- Model output: verbose prose fragments and hallucinated file paths.
- Agreement with rules: 2/10.
- Root cause: the checkpoint was fine-tuned on a raw completion format, not a chat format.

Stage 4 (correct format — `"Return only JSON matching the requested schema.\nTASK_TYPE:…\nJSON:"`):
- Bleed-through eliminated: 10/10.
- Valid JSON schema structure: 10/10.
- But content: 9/10 outputs contain `{"pattern": "<regex>"}` — a literal training-prompt
  placeholder, not a real search term. Only `send_email` (a token from the training repos)
  produced a real identifier.
- A secondary schema bug was discovered: `_schema_valid()` in `slm_runner.py` accepted
  `{"queries": [{"pattern": "..."}]}` as valid because it only checked key presence, not item
  types. This caused the reported "100% first-pass schema validity" to mask content failures.

**Commit:** `fb51cc6` — "fix: schema validator now checks queries item types are strings not dicts"  
**Impact of fix:** 10/10 Stage 4 model outputs now correctly fail validation (previously all passed).

## Stage 5 — Ground-Truth Evaluation: 88.4% Claim Debunked

**Finding:** On 10 ground-truth queries built from real httpx and jinja2 functions, the
search_query_gen model scores 0/10 identifier hits and collapses to placeholder in 8/10 cases.
The 88.4% success figure from the model card is a schema-validation artifact.

Setup:
- Cloned httpx @ `b5addb64`, jinja2 @ `5ef70112`.
- 5 real function names read from `httpx/_client.py`, 5 from `jinja2/environment.py` via AST.
- Correct completion-format prompt applied (Stage 4 fix in place).

Results:
- model\_hits: 0/10
- placeholder\_collapses (output contains `<regex>` literal): 8/10
- rules\_hits: 2/10 (rules at least emit substring matches for `_tokenize` and `_compile`)
- The 88.4% figure relied on the old `_schema_valid()` accepting `[{"pattern": "..."}]` as
  schema-valid. With the fixed validator, all these outputs fail.

**No additional commit** (schema fix from Stage 4 covers this; eval findings documented in Stage 6 below).

## Stage 6 — Context Reduction: Three-Condition Experiment

**Finding:** The deployed rules pipeline achieves 0/10 recall and provides no context reduction.
The oracle upper bound achieves 10/10 recall with 89.7% context reduction.

Setup:
- 10 ground-truth queries across httpx (133,516 tokens, 60 .py files) and jinja2 (183,375
  tokens, 60 .py files).
- tiktoken cl100k_base; grep -r -l fallback (ripgrep not installed on GPU instance).

| Condition | Mean tokens | Reduction vs A | Recall@5 |
|---|---:|---:|:---:|
| A — full repo | 158,446 | — | — |
| B — rules pipeline | 0 | 100%\* | 0/10 |
| C — oracle (ground-truth grep) | 16,394 | 89.7% | 10/10 |

\*100% reduction because the pipeline returned no files at all — not a useful result.

Root cause of B failure: `generate_search_queries_rules()` constructs terms from stopword-
filtered query words (e.g., "find the function that checks if a redirect is HTTPS" →
`checks_if_redirect_https`), which are behavioral descriptions that never match actual Python
identifiers (`_is_https_redirect`). The mismatch is structural: real identifiers are often
shorter, prefixed with underscores, or use different vocabulary than natural-language queries.
