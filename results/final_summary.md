## 1. Starting State — What the Repo Claimed vs. What Actually Ran

The repository presents itself as a working SLM orchestration harness with trained specialist
checkpoints for six developer-agent subroutines: json_repair, trace_localize, search_query,
search_rank, failure_classify, and patch_risk. The benchmark artifacts (`artifacts/results/`)
report 1.0 success rates across most conditions and the whitepaper claims 88.4% success on
a held-out benchmark for search_query generation.

**What the production code paths actually executed (pre-audit):**

- `json_repair` "SLM" path: `_slm_like_json_repair()` — `ast.literal_eval()` + regex, no model
- `trace_localize`: pure regex frame extraction, no model
- `search_query`: stopword-filter + string join, no model
- `search_rank`: regex scoring heuristic, no model
- `failure_classify`: regex pattern matching unless `SLM_HARNESS_FAILURE_CLASSIFIER_MODEL_DIR` set
- `patch_risk`: same pattern as failure_classify

The four HuggingFace checkpoints in `models/huggingface/` were Git LFS pointers (135 bytes each;
1.4–2.0 GB weights not downloaded). No code path in the runtime loaded them. The `slm_only`
and `schema_slm_fallback` labels in `artifacts/results/raw/subroutine_benchmark.jsonl` refer to
deterministic Python, not model inference. The `gpu_memory_mb: 567` in those records came from
a prior Lambda A10 run where `failure_classifier_v1` and `patch_risk_classifier_v1` were loaded
via environment variable — not reproducible from the repo's default state.

## 3. Finding 1 — `failure_classifier_v1` Is Near-Random on Real Inputs

**Commit:** `964ed0b`  
**File:** `results/failure_classifier_v1_analysis.md`

The checkpoint is real BERT-tiny (4.4M params, 17 MB). On 10 hand-written, non-synthetic
failure log snippets run on the A100:

- Model confidence range: 0.128–0.163 (9-class random baseline: 0.111)
- Model accuracy: 2/10
- Deterministic rules accuracy: 8/10 on the same inputs

**Root cause:** Training and evaluation data are both generated by `make_synthetic_examples()`
in `scripts/train_remaining_subroutines.py:364–403`. The factory injects the exact keyword
strings that the regex rules already match (e.g. `f"ModuleNotFoundError: No module named
'rich_{i}'"` for `dependency_issue`). The model achieves 1.0 dev accuracy on a dev set
drawn from the same factory — not on independent data. When presented with real logs that use
varied phrasing, it produces near-uniform logits.

The reported `"device": "cuda", elapsed: 10.4s, accuracy: 1.0"` is accurate as a description
of what happened; the accuracy is a tautology of the data construction, not a signal of
generalization.

## 4. Finding 2 — `search_query_gen` Inference Setup Bug Found and Fixed

**Stages 3–4**

The Qwen2.5-0.5B checkpoint (`slm-agent-search-query-gen-qwen2-5-0-5b`, 1.9 GB, real weights)
was initially run with `tokenizer.apply_chat_template()` using system/user message roles —
the wrong format. The model was SFT'd on a raw completion template (`_render_prompt()` in
`slm_runner.py:515–522`):

```
Return only JSON matching the requested schema.
TASK_TYPE: search_query
SCHEMA: {"properties": {"confidence": ...}, "required": [...]}
INPUT: {"task": "<query>"}
JSON:
```

With the wrong format (Stage 3), the model produced prose fragments and hallucinated file
paths. With the correct format (Stage 4), bleed-through was eliminated and all 10 outputs
were structurally valid JSON — but 9/10 contained `{"pattern": "<regex>"}`, a literal
training-prompt placeholder, not a search term.

## 5. Finding 3 — 88.4% Success Claim Is a Schema-Validation Artifact

**Stage 5 | Commit:** `fb51cc6`

On 10 ground-truth queries built from real httpx and jinja2 functions (AST-verified
identifiers, held-out repos per the model card), the model scored:

- model_hits: 0/10 (no ground-truth identifier present in any output)
- placeholder_collapses (literal `<regex>` in output): 8/10
- The remaining 2 outputs contained the query text slugified to snake_case, not the identifier

The 88.4% figure from the model card (`"Success after one schema-feedback retry"`) was
measured against the old `_schema_valid()` function, which only checked that required keys
**existed** in the output dict — not that their values were correct types. An output of
`{"queries": [{"pattern": "<regex>"}], "confidence": 0.9}` passed as schema-valid because
`"queries"` and `"confidence"` were present, regardless of `queries` containing objects
instead of strings.

---

## 6. Bug Fix — Schema Validator Item-Type Check

**Commit:** `fb51cc6` — "fix: schema validator now checks queries item types are strings not dicts"  
**File:** `src/openharness/orchestration/slm_runner.py:566–582`

**What was wrong:** `_schema_valid()` returned `True` for `{"queries": [{"pattern": "foo"}]}`
when the schema declared `queries.items.type = "string"`. It only verified key presence via
`all(key in output for key in required)`.

**Fix:** After the required-key check, walk array fields against their declared `items.type`;
return `False` if any item is the wrong Python type.

**Impact:** 10/10 Stage 4 model outputs now correctly fail validation (previously all passed).
The "100% first-pass schema validity" claim in the model card is invalidated by this fix.

## 7. Finding 4 — Rules Pipeline Achieves 0/10 Recall

**Stage 6 | Commit:** `0056417`  
**File:** `results/context_reduction_experiment.md`

The deployed `generate_search_queries_rules()` function (`subroutine_models.py:231–248`) was
measured against 10 ground-truth queries on the httpx and jinja2 repos (tiktoken cl100k_base):

- B recall@5: **0/10** — every grep search returned zero file matches
- B mean context tokens: **0** (no files found)
- B "context reduction": 100% (trivially, by finding nothing)

Root cause: the function filters stopwords and joins remaining query words as snake_case, a
4-word regex, and CamelCase. These are behavioral descriptions ("checks_if_redirect_https")
that never match Python identifiers ("_is_https_redirect"). The vocabulary mismatch is
structural.

## 11. Remaining Limitation — 3 Persistent Misses

| Query | ground\_truth | Stemmed query tokens | Stemmed gt tokens | Root cause |
|---|---|---|---|---|
| Q4 | `_merge_queryparams` | `['merg','queri','paramet','request']` | `['merg','queryparam']` | `queryparams` is a fused compound: no underscore, no camelCase boundary, stem produces `queryparam` which doesn't match `queri` or `paramet`. Only `merg` overlaps. |
| Q8 | `_tokenize` | `['token','jinja2','templat','sourc','string']` | `['token']` | Single-token identifier after stemming (`token`). Query tokens `templat`, `sourc`, `string` strongly match multi-token identifiers (`TemplateData`, `from_string`, `get_source`), which outscore `_tokenize`. |
| Q9 | `_compile` | `['compil','templat','string','code','object']` | `['compil']` | Single-token identifier (`compil`). `compile_templates` (rank 3) matches `compil` + `templat`; `from_code` (rank 1) matches `code`. `_compile` (private method) is outscored by its public siblings and multi-token relatives. |

**Common pattern for Q8 and Q9:** private single-token methods (`_tokenize`, `_compile`) cannot
compete against multi-token siblings when the query is multi-term. A docstring-aware retrieval
or definition-site boosting (prefer `def _compile` over `compile` and `compile_templates`)
would address this.

**Q4** requires splitting fused compound identifiers before indexing — e.g., a dictionary-
based decomposition of `queryparams` → `["query", "params"]`.

## 12. Research Direction

An SLM for this task would need to learn a mapping from natural-language intent ("find the
method that merges query parameters") to Python identifier tokens that appear in the target
codebase. The correct training data is pairs of `(NL query, ground-truth identifier)` derived
from real repositories, where the identifier is found by static analysis — not a synthetic
factory. The correct evaluation metric is recall@k of the ground-truth identifier in the
retrieved file set, measured on held-out repos not seen during training. The correct baseline
to beat is BM25+stem with docstring augmentation and compound-split preprocessing, which
achieves 7/10 recall@5 and 85.8% context reduction with no model and no GPU. An SLM adds
value only on queries where surface-form token matching fails: private single-token methods
(`_compile`, `_tokenize`) and fused compound identifiers (`queryparams`) where the query
describes the behavior but not the name. That is a narrow, well-defined target: roughly 30%
of queries in this evaluation. The SLM would need to achieve >7/10 recall@5 on these
specifically hard queries — not on easy queries where BM25 already wins — to justify the
inference overhead.

## 13. Scaled, GPU-Backed Retrieval Study (supersedes the n=10 story)

The n=10 proof-of-concept was scaled to a real benchmark and measured against strong lexical
and neural baselines. Full detail: `results/retrieval_eval_full.md`.

**Eval set:** 1,406 auto-generated ground-truth queries across 6 pinned repos (httpx, jinja2,
requests, flask, click, rich). Each query is the first docstring sentence of a function/class
with identifier-name tokens stripped; ground truth = identifier + file.
(`evals/context_retrieval_queries.jsonl`, generator `scripts/build_context_retrieval_queries.py`.)

**Results (full set, recall@5 / mean context reduction):**

| Method | recall@1 | recall@5 | reduction | notes |
|---|---:|---:|---:|---|
| BM25+stem baseline | 0.780 | **0.966** | 86.9% | winner overall, deterministic, no GPU |
| Improved lexical (compound+def-site) | 0.723 | 0.942 | 87.8% | **wins on impl subset: 0.977 r@5** |
| Neural (jina-embeddings-v2-base-code, 160.9M) | 0.421 | 0.627 | 87.9% | real GPU inference, 1,581 emb/s, 3,541 MiB |
| Hybrid RRF | 0.289 | 0.790 | 85.9% | lost to lexical; dropped, not integrated |

**Key findings (all honest, including the negatives):**
- The plain **BM25+stem lexical baseline wins** (r@5 0.966) and is hard to beat. At scale,
  name+docstring BM25 is genuinely strong — the n=10 set (7/10) understated it.
- The **improved retriever wins on the implementation-target subset** (impl r@5 0.962→0.977,
  better at every k) but its test-file penalty costs recall on the 360 test-target queries,
  dragging the full-set number down 2.3pp. This is the correct heuristic for finding
  implementation, reported with its tradeoff.
- The **neural model lost to lexical** by ~34pp at r@5. Caveat: queries are docstring-derived,
  which favors lexical surface overlap; this is not a general verdict against neural retrieval.
- The **hybrid did not beat both singles** and was dropped per plan.
- A **new module** `src/openharness/orchestration/code_retriever.py` ships the lexical retriever
  behind an opt-in MCP tool `slm_code_context_search` (gated by
  `SLM_HARNESS_ENABLE_CODE_RETRIEVER`); default behavior unchanged. 10 unit tests pass; ruff clean.
- **Workstream E (downstream answer-preservation) was SKIPPED** — no API key present; no
  downstream numbers fabricated (`results/downstream_value_test_SKIPPED.md`).

The earlier negative findings about the existing checkpoints (Sections 3–11) stand unchanged:
`failure_classifier_v1` is near-random on real inputs, `search_query_gen` collapses to a
placeholder and its 88.4% was a schema-validation artifact (since fixed).
